The 48-digit LUKS recovery key generated by `phase-2-arch-install/install.sh` is stored in Bitwarden in a human-readable hyphenated format (BitLocker convention: typically 8 groups of 6 with `-` separators).

**Why:** install.sh prints the key with hyphens for human readability when Tom photographs it, and the bw entry mirrors that format. cryptsetup / systemd-cryptenroll are byte-strict — they don't strip formatting, so `123456-789012-...` in `--unlock-key-file` fails to unlock while the same 48 digits as `123456789012...` succeeds.

**How to apply:** whenever fetching the LUKS recovery key from bw for `cryptsetup` / `systemd-cryptenroll --unlock-key-file=`, always pipe through `tr -cd '0-9'`. Sanity-check that the resulting file is exactly 48 bytes before using it (`wc -c`). If it isn't 48, do not proceed — the bw entry has been edited or the wrong record was matched.
